  GRUPO 1 – EXCEPCIONES DEL DOMINIO (test_exceptions.py) | Tests T01–T06
================================================================================

T01 – test_T1_broken_tunes_error_es_exception
  Archivo : tests/test_exceptions.py
  Clase   : (módulo)
  Objetivo: Verificar que BrokenTunesError hereda directamente de la clase base
            Exception de Python.
  Razón   : Al ser subclase de Exception, cualquier capa del sistema puede
            capturar errores del dominio con "except BrokenTunesError" sin
            necesitar conocer los tipos concretos.
  Técnica : issubclass(BrokenTunesError, Exception)

──────────────────────────────────────────────────────────────────────────────

T02 – test_T2_song_not_found_error_hereda_de_broken_tunes_error
  Archivo : tests/test_exceptions.py
  Objetivo: Verificar que SongNotFoundError hereda de BrokenTunesError.
  Razón   : La jerarquía de herencia permite que un manejador genérico de errores
            del dominio capture todas las excepciones propias sin conocerlas
            individualmente.
  Técnica : issubclass(SongNotFoundError, BrokenTunesError)

──────────────────────────────────────────────────────────────────────────────

T03 – test_T3_song_not_found_error_atributo_y_mensaje
  Archivo : tests/test_exceptions.py
  Objetivo: Verificar que SongNotFoundError(99) almacena el id en .song_id
            y que el mensaje str() de la excepción contiene "99".
  Razón   : Los manejadores de errores deben poder extraer el id sin parsear
            el mensaje de texto; y el mensaje debe identificar el recurso
            fallido para facilitar el debugging.
  Técnica : Instanciar SongNotFoundError(99), assert error.song_id == 99,
            assert "99" in str(error)

──────────────────────────────────────────────────────────────────────────────

T04 – test_T4_backup_not_found_error_atributo_y_mensaje
  Archivo : tests/test_exceptions.py
  Objetivo: Verificar que BackupNotFoundError(42) es subclase de BrokenTunesError,
            almacena .backup_id == 42 y contiene "42" en el mensaje.
  Razón   : Análogo a T03, pero para backups. Garantiza consistencia en la
            jerarquía y en el acceso a metadatos del error.
  Técnica : issubclass + atributo + contenido del mensaje

──────────────────────────────────────────────────────────────────────────────

T05 – test_T5_empty_audio_error_incluye_nombre_recurso
  Archivo : tests/test_exceptions.py
  Objetivo: Verificar que EmptyAudioError("canción id=7") es subclase de
            BrokenTunesError y que el mensaje incluye el nombre del recurso.
  Razón   : El nombre del recurso en el mensaje permite a los logs identificar
            exactamente qué elemento no tenía datos de audio.
  Técnica : issubclass + assert resource in str(error)

──────────────────────────────────────────────────────────────────────────────

T06 – test_T6_duplicate_song_error_incluye_titulo_y_artista
  Archivo : tests/test_exceptions.py
  Objetivo: Verificar que DuplicateSongError("Bohemian Rhapsody", "Queen")
            es subclase de BrokenTunesError y que el mensaje incluye ambos
            el título y el artista.
  Razón   : El usuario o el log debe saber cuál combinación (título+artista)
            causó la violación de clave única.
  Técnica : issubclass + assert title in str(error) + assert artist in str(error)


================================================================================
  GRUPO 2 – CONFIGURACIÓN (test_config.py) | Tests T07–T11
================================================================================

T07 – test_T7_db_params_devuelve_dict
  Archivo : tests/test_config.py
  Objetivo: Verificar que Config.db_params() retorna un objeto de tipo dict.
  Razón   : mysql-connector-python espera un diccionario como parámetros de
            conexión. Si el tipo fuera incorrecto, la conexión fallaría en
            tiempo de ejecución con un error poco descriptivo.
  Técnica : isinstance(Config.db_params(), dict)

──────────────────────────────────────────────────────────────────────────────

T08 – test_T8_db_params_contiene_claves_requeridas
  Archivo : tests/test_config.py
  Objetivo: Verificar que el dict de Config.db_params() contiene las claves
            "host", "user", "password" y "database".
  Razón   : Estas son exactamente las claves que mysql-connector-python requiere.
            Si faltara alguna, la conexión lanzaría un TypeError en tiempo de
            ejecución, difícil de rastrear.
  Técnica : set.issubset sobre el resultado de db_params()

──────────────────────────────────────────────────────────────────────────────

T09 – test_T9_debug_por_defecto_es_false
  Archivo : tests/test_config.py
  Objetivo: Verificar que Config.DEBUG es de tipo bool y es False cuando
            la variable de entorno FLASK_DEBUG no está definida.
  Razón   : Correr Flask en modo debug en producción expone el debugger
            interactivo de Werkzeug, lo que es una vulnerabilidad crítica.
            El default seguro es False.
  Técnica : isinstance(Config.DEBUG, bool) + comparación condicional

──────────────────────────────────────────────────────────────────────────────

T10 – test_T10_host_por_defecto_es_0000
  Archivo : tests/test_config.py
  Objetivo: Verificar que Config.HOST == "0.0.0.0" cuando FLASK_HOST no está
            definido en el entorno.
  Razón   : "0.0.0.0" permite que el servidor acepte conexiones desde cualquier
            interfaz, necesario para entornos Docker o CI donde el servidor
            debe ser accesible externamente.
  Técnica : Comparación directa si os.getenv("FLASK_HOST") is None

──────────────────────────────────────────────────────────────────────────────

T11 – test_T11_port_por_defecto_es_5000_y_es_entero
  Archivo : tests/test_config.py
  Objetivo: Verificar que Config.PORT es de tipo int con valor 5000 por defecto.
  Razón   : Flask.run() requiere un entero para el puerto. Si fuera un string
            "5000" lanzaría TypeError al iniciar la aplicación.
  Técnica : isinstance(Config.PORT, int) + comparación condicional

================================================================================
  CÓMO EJECUTAR LAS PRUEBAS Y GENERAR EL PDF
================================================================================

1. Instalar dependencias:
   pip install pytest reportlab

2. Ejecutar solo los tests (desde Backend/):
   pytest tests/ -v

3. Ejecutar tests Y generar el PDF del reporte (desde Backend/):
   python generate_report.py

4. El PDF se guarda en:
   Backend/reporte_pruebas.pdf
